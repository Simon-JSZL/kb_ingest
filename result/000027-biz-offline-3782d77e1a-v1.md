## 1. 核心内容

B.4 生产运维信息查询报文(计划时间段查询) 用于基于计划时间段对生产运维信息进行查询。报文整体采用 JSON 格式，分为请求报文结构和应答报文结构。

**请求报文结构**包含 `request` 根节点，其下分为 `head`、`body` 和 `signature` 三个部分：
- **head（报文头）**：包含 `sysId`（取值为 `uops`）、`msgId`（取值为 `uops031.000.01`）、`sendDate`（格式为 `YYYY-MM-DD HH:mm:ss`）、`senderInstId`（发起方机构标识）、`receiverInstId`（接收方机构标识）、`direction`（取值为 `01`）。
- **body（报文体）**：包含 `startTime`（查询开始时间）、`endTime`（查询结束时间）、`infoNo`（在计划时间段查询场景下，该字段传空字符串 `""`）。
- **signature（数字签名域）**：内容为"签名内容"。

**应答报文结构**包含 `response` 根节点，其下同样分为 `head`、`body` 和 `signature` 三个部分：
- **head（报文头）**：除了包含与请求报文相同的 `sysId`（`uops`）、`msgId`（`uops031.000.01`）、`sendDate`（`YYYY-MM-DD HH:mm:ss`）、`senderInstId`（发起方机构标识）、`receiverInstId`（接收方机构标识）、`direction`（`01`）外，还包含应答专属字段 `rspCd`（应答代码，示例值为 `0000`）和 `rspMsg`（应答消息）。
- **body（报文体）**：包含查询返回的详细信息，字段有 `infoNo`（信息编号）、`instName`（发起方单位名称）、`instId`（发起方金融编码）、`reinstName`（接收方单位名称）、`reinstId`（接收方金融编码）、`startTime`（开始时间）、`endTime`（结束时间）、`reason`（原因）、`contactPerson`（联系人）、`contactPhone`（联系电话）。
- **signature（数字签名域）**：内容为"签名内容"。

## 2. 适用边界

本报文结构适用于网络支付清算平台联合运维场景下的生产运维信息查询，具体用于"计划时间段查询"场景。与"运维信息编号查询"场景共用 `msgId` 为 `uops031.000.01` 的报文，区别在于请求报文体的 `infoNo` 字段：在本计划时间段查询场景中，请求体的 `infoNo` 字段需设置为空字符串 `""`，同时提供 `startTime` 和 `endTime`。

## 3. 使用要求

- 发起查询请求时，必须严格遵循 JSON 结构组装报文。
- 请求报文头中的 `sendDate` 必须符合 `YYYY-MM-DD HH:mm:ss` 格式要求。
- 在计划时间段查询场景下，请求体中的 `infoNo` 必须为空字符串 `""`，查询条件依赖于 `startTime`（查询开始时间）和 `endTime`（查询结束时间）。
- 请求报文和应答报文中均包含 `signature`（数字签名域），其内容为"签名内容"。
- 应答处理时，需通过应答报文头中的 `rspCd`（如 `0000`）和 `rspMsg` 判断请求处理状态，并从应答报文体中提取相关信息。
