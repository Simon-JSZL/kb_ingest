# 通用知识库建立规范

## 1. 建设目标

知识库用于把权威资料整理成可检索、可回答、可追溯的独立知识条目。规范的首要目标是准确、完整保留来源正文事实，而不是把所有材料强行套进某个业务模板。

核心目标：

1. 每个条目围绕一个清晰知识点，可独立回答用户问题。
2. 除格式、页码、页眉页脚、目录噪声等非内容信息外，正文应全量覆盖来源中的定义、条件、规则、流程、阈值、例外和限制。
3. 没有出处的信息不写；不确定的信息标明“原文未明确说明”。
4. 条目结构保持稳定，但允许不同资料类型自然表达，不强行填业务场景、角色或处置动作。
5. 每个条目必须能追溯到来源文档、章节和版本。

## 2. 条目类型

`doc_type` 用于辅助检索和过滤，必须依据原文内容选择。无法明确归类时使用 `biz`。

| 类型 | 说明 |
| --- | --- |
| `biz` | 业务、系统、对象、概念、范围或通用说明 |
| `scenario` | 场景、状态、事件、问题表现或适用情形 |
| `sop` | 操作步骤、处理流程、执行要求 |
| `metric` | 指标定义、统计口径、计算方式、阈值 |
| `severity` | 分级、定级、优先级或等级判断 |
| `change` | 变更、发布、评估、通知或回退要求 |
| `function` | 可调用能力、工具、接口或函数说明 |
| `evaluation` | 评价对象、评价周期、评价指标、评价结果 |

## 3. 单篇知识文档格式

每个知识条目使用 Markdown 文件保存，并使用 YAML Front Matter 描述元数据。正文固定为 5 个二级章节，减少空槽位，优先保证事实完整。

```markdown
---
kb_id: "biz-source-topic-v1"
title: "知识条目标题"
doc_type: "biz"
domain: "知识库领域"
category: "来源资料或知识大类"
category_keywords: ["关键词1", "关键词2"]
subcategory: "知识小类或场景标题"
related_items:
  - 大类标题: "来源资料或知识大类"
    小类标题: "关联小类标题"
    关联说明: "说明为什么相关"
    关联度: "高"
business_modules: []
source_doc: "来源文档.md"
source_version: "V1.0"
source_section: "3.1 来源章节"
effective_date: ""
owner: "知识库维护方"
confidentiality: "内部"
risk_level: "low"
applicable_roles: []
tags: ["标签1", "标签2"]
status: "active"
---

# 标题

## 1. 核心内容

## 2. 适用边界

## 3. 使用要求

## 4. 关联能力

## 5. 来源依据
```

## 4. 元数据规范

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `kb_id` | 是 | 全局唯一 ID，使用英文、数字和短横线 |
| `title` | 是 | 条目标题，应直接表达该条目回答什么问题 |
| `doc_type` | 是 | 条目类型，取值见“条目类型” |
| `domain` | 是 | 所属知识库领域 |
| `category` | 是 | 知识大类，通常来自源文件大标题、首页标题或文档主题 |
| `subcategory` | 是 | 知识小类或当前条目场景标题 |
| `category_keywords` | 是 | 关键词一览，围绕小类标题和正文核心内容抽取，不超过 10 个 |
| `related_items` | 是 | 关联信息列表，每项包含大类标题、小类标题、关联说明、关联度 |
| `business_modules` | 是 | 原文明确出现的业务、系统、模块或对象；没有则为空数组 |
| `source_doc` | 是 | 来源文档文件名 |
| `source_version` | 是 | 来源版本；未知可留空 |
| `source_section` | 是 | 来源章节 |
| `effective_date` | 否 | 生效日期；未知可留空 |
| `owner` | 是 | 维护责任方 |
| `confidentiality` | 是 | 密级 |
| `risk_level` | 是 | 回答或使用该知识可能涉及的风险，默认 `low` |
| `applicable_roles` | 是 | 原文明确出现的适用角色；没有则为空数组 |
| `tags` | 是 | 条目级关键词、简称、缩写、对象名 |
| `status` | 是 | 状态 |

`risk_level` 建议枚举：

| 值 | 含义 |
| --- | --- |
| `low` | 解释、查询、制度说明、定义说明 |
| `medium` | 需要判断条件、组合信息或给出建议 |
| `high` | 涉及生产动作、变更动作、降级、暂停、回退等高影响建议 |
| `critical` | 涉及不可逆动作、敏感操作或必须人工审批的内容 |

## 5. 正文内容规范

### 5.1 核心内容

写清该条目最重要、最可回答的事实。根据来源类型自然组织，不强行套业务模板。对于短句、定义句、简称句、阈值句、禁止句和规则句，优先保留原句或等价完整表述，不能压缩到丢失主体、边界或限定语。

必须保留：

1. 术语、定义、全称、简称、英文名、英文简称。
2. 规则、条件、判断逻辑、优先级、例外。
3. 阈值、单位、时间、数量、比例、比较符和适用对象。
4. 流程步骤、输入输出、字段含义、结构关系。

若原文出现“以下简称”“英文简称”“称为”“是指”“定义为”等表达，本节必须写清被定义对象、完整定义句主体、中文简称、英文名和英文简称。不得只放在标签、检索词或来源依据中。

### 5.2 适用边界

说明该知识适用于什么对象、场景、范围、前提条件和不适用范围。原文没有边界时写“原文未明确说明”，不要扩展推断。

### 5.3 使用要求

说明使用该知识回答或执行时需要注意什么。根据原文选择合适内容：

1. 回答要求：必须引用哪些事实、缺哪些信息时如何说明。
2. 操作要求：谁执行、何时执行、按什么顺序执行。
3. 判断要求：先判断什么，再判断什么，是否需要人工确认。
4. 禁止事项：原文明确禁止、限制或需审批的内容。

没有使用要求时写“暂无”或“原文未明确说明”。

### 5.4 关联能力

只记录与当前条目明确相关的查询函数、接口、工具或系统能力。没有明确相关能力时写“暂无”。该章节可由程序根据工具维护文件补写，模型不得编造工具。

### 5.5 来源依据

记录来源文档、来源章节、页码和归纳说明。来源依据只写溯源信息，不承载正文核心事实。

## 6. 切分原则

按“一个可回答知识点一条”切分，而不是按固定业务模板切分。

优先独立成条的内容：

1. 一个术语或定义。
2. 一组紧密相关的规则或条件。
3. 一个流程或操作要求。
4. 一个指标口径或阈值表。
5. 一个分级或评价标准。
6. 一个接口、函数或工具说明。

不要为了凑栏目把无关内容合并，也不要把整份制度原文作为一个向量文档直接入库。

## 7. 质量校验规范

每个条目交付前至少检查：

1. 正文核心内容是否回答了标题所表达的问题。
2. 来源正文事实是否全量覆盖；定义、简称、英文简称、阈值、比较符、单位、时间和适用对象是否保留。
3. 原文没有的信息是否没有被编造。
4. 适用边界是否清楚，缺失时是否明确写“原文未明确说明”。
5. 关联能力是否来自工具维护文件或原文明确描述。
6. 来源依据是否可追溯到文档和章节。
7. 是否存在只有标题、标签或检索词包含关键事实，而正文没有关键事实的情况。
